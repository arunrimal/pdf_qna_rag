Technical Report · 2025

RAG-Based PDF Q&A
Application

End-to-end Retrieval-Augmented Generation system — from PDF ingestion to streaming answers.

LlamaIndex Gemini 2.5 Flash Pinecone FastAPI Next.js Vercel + Render

1System Architecture

The application follows a Retrieval-Augmented Generation (RAG) pattern: rather than asking an LLM to answer from training knowledge alone, the system first retrieves relevant text chunks from the uploaded document and injects them as context into the prompt. This grounds the response in the actual document content.

[User] → Upload PDF
            ↓
       [FastAPI Backend]
            ↓
       Text Extraction (PyMuPDF + LlamaIndex SimpleDirectoryReader)
            ↓
       Chunking (LlamaIndex default node parser)
            ↓
       Embedding (Gemini gemini-embedding-001, 768-dim vectors)
            ↓
       Vector Storage (Pinecone, namespace = session_id)
            ↓
[User] → Ask Question
            ↓
       Query Embedding (same Gemini model)
            ↓
       Semantic Search (Pinecone cosine similarity, top-k nodes)
            ↓
       Context Assembly (retrieved chunks + chat history)
            ↓
       LLM Generation (Gemini 2.5 Flash)
            ↓
       Streaming Response (SSE tokens) → [Next.js Frontend]

1.1 Ingestion Pipeline

When a user uploads a PDF:

  1. Receive & validate: The /upload endpoint accepts a multipart/form-data request. PyMuPDF (fitz) opens the file and counts pages — uploads exceeding 5 pages are rejected with HTTP 400.
  2. Save temporarily: The file is written to a system temp directory (tempfile.mkdtemp()).
  3. Text extraction: SimpleDirectoryReader(input_files=[pdf_path]).load_data() parses the PDF into LlamaIndex Document objects. Each document carries metadata: page_label and file_name.
  4. Cleanup: Temp files are deleted immediately after processing to avoid disk accumulation.

1.2 Chunking

LlamaIndex's default node parser breaks Document objects into smaller Node objects (text chunks). These nodes preserve parent-document metadata and are the unit of embedding and retrieval. The chunking strategy uses LlamaIndex's internal defaults (typically ~1024 token chunks with overlap).

1.3 Embedding

Each node's text is passed to Google's gemini-embedding-001 model via GoogleGenAIEmbedding, producing a 768-dimensional dense vector. These vectors encode the semantic meaning of each chunk. Embedding is triggered automatically when VectorStoreIndex.from_documents() is called.

1.4 Vector Storage

Embeddings are stored in Pinecone, a managed vector database. Each user session gets its own Pinecone namespace (keyed by session_id), providing logical data isolation.

vector_store = PineconeVectorStore(
    pinecone_index=pinecone_index,
    namespace=session_id   # Per-user isolation
)

1.5 Retrieval

When the user asks a question:

  1. The query is embedded with the same Gemini model.
  2. Pinecone performs cosine similarity search within the user's namespace.
  3. The top-k most semantically similar nodes are returned.
  4. LlamaIndex assembles these nodes into a context window alongside the conversation history from ChatMemoryBuffer (capped at 3,900 tokens).

1.6 Generation

The assembled context + query are passed to Gemini 2.5 Flash via LlamaIndex's GoogleGenAI wrapper. The chat engine operates in context mode with a system prompt that instructs the model to treat the PDF as ground truth for facts, while allowing creative latitude when the user asks to generate new content (e.g., a cover letter). Responses are streamed token-by-token using Server-Sent Events (SSE).

3Challenges Faced

Development Challenges

Challenge 1 — Sync/Async Mismatch in LlamaIndex

FastAPI is async-first, but LlamaIndex's ChatEngine.chat() and stream_chat() are synchronous. Calling a sync blocking function directly from an async endpoint blocks the event loop, breaking concurrent request handling.

Solution:

loop = asyncio.get_event_loop()
response = await loop.run_in_executor(None, lambda: engine.chat(query))

This offloads the blocking call to a worker thread while the event loop remains free to handle other requests.

Challenge 2 — Streaming with Sync Generators

engine.stream_chat() returns a synchronous generator (response_gen). Converting this to an async SSE stream required wrapping it in an async def event_generator() that iterates the sync generator and yields SSE-formatted strings.

Challenge 3 — Session State Management

There is no database for session state — chat engines are stored in-process memory (active_sessions dict). This creates simplicity but also risk: a server restart wipes all active sessions. The Pinecone embeddings persist, but the ChatMemoryBuffer (conversation history) is lost.

Challenge 4 — Source Metadata Parsing Fragility

LlamaIndex's source node format is not fully stable across versions. The source extraction code has multiple fallback paths (try/except) because the attributes (node, text, metadata) were not always present in the same location across different response types.

Deployment Challenges

Challenge 5 — Cold Starts on Render Free Tier

Render's free tier spins down containers after 15 minutes of inactivity. The first request after a cold start can take 30–60 seconds, which appears to the user as a hang.

Mitigations:

  • Upgrade to a paid Render tier (always-on).
  • Use an uptime monitoring service (e.g., UptimeRobot) to ping the backend every 10 minutes.
Challenge 6 — CORS Configuration

The FastAPI backend must explicitly allow the Vercel frontend origin. Misconfigured CORS silently fails in the browser with an opaque network error. The current implementation uses a permissive CORS policy (allow_origins=["*"]) acceptable for a demo but must be tightened for production.

Challenge 7 — Environment Variable Management

Vercel uses NEXT_PUBLIC_ prefixed variables for client-side exposure. The backend URL must be set as NEXT_PUBLIC_API_URL. Forgetting the prefix means the variable is server-side only and unavailable to browser fetch calls.

Challenge 8 — Pinecone Index Initialization Race Condition

The index creation code checks if INDEX_NAME not in pc.list_indexes().names() before creating. On first deployment, the index may not be fully ready when the first request arrives. The wait_until_ready=True flag mitigates this but adds startup latency.

4Limitations

RAG-Specific Limitations

  1. Retrieval errors (the core RAG failure mode)
    The system can only answer from chunks that were successfully retrieved. If the relevant information is poorly chunked, semantically distant from the query phrasing, or beyond the top-k cutoff, the answer will be wrong or incomplete — even though the document contains the answer.

  2. Hallucination at boundaries
    When retrieved context is insufficient, Gemini 2.5 Flash may fill gaps with plausible-sounding but fabricated content. The system prompt instructs the model to "stick strictly to context," but LLMs can still confabulate when context is thin.

  3. Chunk boundary problems
    If a key piece of information spans two chunks, neither chunk alone may score high enough on retrieval. The answer is "in" the document but the retrieval system cannot surface it cleanly.

  4. Context window limits
    The 3,900-token ChatMemoryBuffer means that in long conversations, early exchanges are dropped, potentially losing important context.

  5. Single-document restriction
    The current architecture associates one session with one PDF. Cross-document Q&A is not supported.

  6. 5-page limit
    Substantial documents routinely exceed 5 pages. This is a practical cost constraint but significantly restricts real-world utility.

Tool / Framework Limitations

  1. LlamaIndex abstraction opacity
    LlamaIndex simplifies the RAG pipeline but hides internals. Debugging retrieval quality or tuning similarity thresholds requires reaching past the abstraction.

  2. No re-ranking step
    The system uses raw embedding similarity for retrieval. There is no cross-encoder reranker to refine the top-k results.

  3. Pinecone free tier constraints
    Heavy usage or large PDFs will exhaust quotas and cause failures.

  4. Gemini API rate limits
    High traffic causes 429 errors with no built-in retry logic in the current implementation.

  5. Ephemeral session state
    active_sessions lives in process memory with no TTL or eviction policy. Long-running deployments accumulate memory from sessions never explicitly cleared.

5Improvements & Future Work

Technical Improvements

1. Add a Reranker

After initial vector retrieval, pass the top-k nodes through a cross-encoder reranker (e.g., Cohere Rerank, or a local cross-encoder/ms-marco-MiniLM model). Rerankers use both the query and the chunk together for scoring, yielding much higher retrieval precision than embedding similarity alone.

2. Hybrid Retrieval (Semantic + Keyword)

Combine dense vector search (current) with BM25 keyword search. Pinecone supports sparse-dense hybrid queries. This handles cases where exact keyword matches (names, IDs, technical terms) outperform semantic similarity.

3. Sentence Window Retrieval

Store embeddings at the sentence level but retrieve the surrounding window (e.g., 3 sentences before/after). This gives fine-grained retrieval precision without losing surrounding context needed for coherent answers.

4. Persistent Session Storage

Replace the in-memory active_sessions dict with Redis. This enables survival across server restarts, TTL-based session expiration, and horizontal scaling.

5. Explicit Chunk Size Tuning

from llama_index.core.node_parser import SentenceSplitter
node_parser = SentenceSplitter(chunk_size=512, chunk_overlap=64)

6. Evaluation Pipeline

Implement automated RAG evaluation using RAGAS or TruLens, measuring Faithfulness, Answer Relevancy, and Context Recall.

Scalability & Production Readiness

  1. Authentication & authorization — Add JWT-based auth. Session IDs should be tied to authenticated users.
  2. File validation — MIME type checking, file size limits (10 MB max), and virus scanning.
  3. Queue-based ingestion — Move PDF processing to a background job queue (Celery + Redis or AWS SQS). The /upload endpoint returns a job ID immediately.
  4. Observability — Structured logging with session_id, query, latency_ms, tokens_used.
  5. Cost controls — Per-session token caps and Redis-based rate limiting per user.
  6. Query caching — Cache embeddings for repeated queries using query hash as the Redis key.
  7. Upgrade hosting — Google Cloud Run with min-instances=1 or AWS ECS Fargate to eliminate cold-start latency.

6Industry Challenges in RAG Systems

Retrieval Quality

Embedding space collisions: Two semantically different sentences can produce similar embeddings, causing irrelevant chunks to score higher than relevant ones with no way to detect it at runtime.

Query-document mismatch: User queries are often short and conversational; indexed documents are long and formal. Query expansion and HyDE (Hypothetical Document Embedding) address this.

Top-k sensitivity: Too few chunks = missing relevant content. Too many = diluting the context with noise, inflating token costs, and degrading generation quality.

Latency & Cost

Embedding latency: Every query requires an embedding API call before retrieval. At scale, this adds 100–300 ms per request. Caching embeddings for repeated queries mitigates this.

Generation latency: Large context windows slow LLM generation. Streaming masks this perceptually, but time-to-first-token is still a user experience factor.

Cost at scale: Fine-tuning a smaller open-source embedding model (e.g., bge-small-en) for self-hosted embeddings can reduce cost by 10–100x.

Data Freshness

Static index problem: Once a PDF is indexed, the embeddings are fixed. If the source document is updated, old embeddings remain stale. Production systems need index update pipelines with document versioning.

Hallucination & Faithfulness

Confident wrong answers: LLMs generate fluent text even when retrieved context is insufficient. Without a faithfulness layer, users cannot distinguish grounded answers from fabrications.

Instruction-following fragility: "Answer only from the document" is a soft constraint. Adversarial queries or edge cases can cause the model to override the instruction.

Evaluation Difficulty

No ground truth: For most document Q&A tasks, there is no labeled dataset of question-answer pairs. Building an evaluation set requires human annotators or LLM-as-judge (which introduces its own biases).

Distribution shift: A RAG system that works on test PDFs may fail on user-uploaded PDFs with different formatting, languages, or domains.

Chunking Sensitivity

The "right" chunk size does not exist: Optimal chunk size varies by document type. A single fixed chunk size is always a compromise.

Table and figure blindness: Standard text chunkers ignore tables, figures, and charts. A PDF where key numbers are in a table will produce chunks that either omit the table entirely or include it as unstructured text the embedding model cannot reliably interpret.

7Interview Preparation — Questions & Answers

Conceptual Questions
Q1What is RAG and when would you use it instead of fine-tuning?

RAG (Retrieval-Augmented Generation) is a pattern where, instead of relying solely on an LLM's training knowledge, you retrieve relevant external documents at inference time and inject them as context into the prompt. The model then generates an answer grounded in that retrieved content.

Use RAG when:

  • Your knowledge changes frequently (new documents, updated policies). You can re-index without retraining.
  • You need source attribution — you can cite the exact chunks used.
  • Your data is private and can't be in training data.
  • You need to support multiple knowledge domains simultaneously.
  • You have limited budget (no GPU training runs).

Use fine-tuning when:

  • You need a specific output style, tone, or format that RAG can't teach through prompting.
  • The knowledge is stable over time and doesn't change.
  • Latency is critical — there's no retrieval step, so inference is faster.
  • You need to teach the model a new capability (not just knowledge), like a new reasoning pattern.

The key distinction: RAG gives the model access to external knowledge at runtime; fine-tuning bakes knowledge into the model weights at training time. For a PDF Q&A app, RAG is clearly correct — the documents are user-provided and change with every upload.

Q2How does your chunking strategy affect retrieval quality?

Chunking is one of the most impactful — and most underappreciated — decisions in a RAG pipeline.

  • Chunk size too small (e.g., 128 tokens): High retrieval precision but each chunk lacks surrounding context. The LLM receives a fragment without enough information to synthesize a complete answer.
  • Chunk size too large (e.g., 2048 tokens): Each chunk contains a lot of context, but the embedding averages over a large piece of text. A chunk about both "Project A" and "Project B" may rank low for a query about "Project A" because the embedding is diluted.
  • Chunk overlap: Prevents information loss at boundaries. If a key sentence falls at the end of chunk 1 and the beginning of chunk 2, overlap ensures at least one chunk captures it cleanly.

In this project, LlamaIndex defaults (~1024 tokens) are used. For a resume or short PDF, smaller chunks (256–512 tokens) would work better since information is dense and query-specific. The ideal approach is to tune chunk size experimentally using retrieval evaluation metrics.

Q3What is the difference between sparse and dense retrieval? When would you use BM25 over embeddings?
  • Dense retrieval (what this project uses): Encodes both query and documents as continuous vectors. Captures semantic similarity — "automobile" matches "car" even though the words differ.
  • Sparse retrieval (BM25/TF-IDF): Represents documents as bags of words with weighted frequencies. Captures exact lexical overlap.

Use BM25 (or hybrid) when: Queries contain specific identifiers — names, product IDs, error codes, technical terms, dates. Dense embeddings may blur these specifics. Example: "What is the error code E404 handling?" — BM25 will exactly match "E404"; embeddings might retrieve generic error-handling sections.

Best practice in production: Use hybrid retrieval — a weighted combination of both scores. Pinecone supports this natively. Most industry RAG systems outperform pure dense or pure sparse retrieval with a hybrid approach.

Q4How do you evaluate a RAG system? What metrics do you use?

RAGAS (Retrieval Augmented Generation Assessment):

  • Faithfulness: Are all claims in the generated answer supported by the retrieved context? (Measures hallucination)
  • Answer Relevancy: Does the answer actually address the question?
  • Context Precision: Of the retrieved chunks, how many were actually relevant? (Measures retrieval noise)
  • Context Recall: Were all the relevant chunks retrieved? (Measures retrieval completeness)

TruLens: Wraps the full RAG pipeline and evaluates each step (retrieval, generation) using an LLM-as-judge.

For this project specifically, you could evaluate by creating a set of (question, expected answer, page number) triples for a test PDF, running them through the pipeline, and comparing retrieved sources and answers.

Q5What is HyDE and when is it useful?

HyDE stands for Hypothetical Document Embedding. Instead of embedding the raw user query and searching for similar chunks, HyDE works in two steps:

  1. Ask the LLM to generate a hypothetical answer to the query — a short paragraph that looks like it could be from the document.
  2. Embed that hypothetical answer and use it as the search vector.

Why this works: A short user query like "What are John's skills?" may not embed close to the chunk in a resume that says "Proficient in Python, React, and SQL." But a hypothetical answer — "John is proficient in Python, React, and SQL" — will embed very close to that chunk.

When to use HyDE: Queries are short, vague, or conversational while documents are formal and dense. When direct query embedding gives poor retrieval recall.

Tradeoff: HyDE adds an extra LLM call before retrieval, increasing latency and cost. Not worth it for simple factual queries where the query and document phrasing naturally align.

Q6How would you handle a PDF with tables and figures?

Standard text extraction (PyMuPDF + LlamaIndex defaults) treats tables as flat text strings or skips them entirely, which breaks semantic retrieval for numeric/tabular data.

For tables: Use a specialized parser — LlamaParse, Azure Document Intelligence, or Unstructured.io — which extract tables as markdown or structured JSON. Store each table as a separate node with metadata indicating it's a table.

For figures/charts: Use a multimodal embedding model (e.g., GPT-4 Vision, Gemini Vision) to generate a text caption for each figure. Store the caption as a node linked to the figure's location in the PDF.

For complex PDFs (mixed content): Use a layout-aware parser that understands reading order, column layouts, and headers — standard PDF text extraction reads left-to-right without layout context and produces garbled text from multi-column documents.

System Design Questions
Q7How would you scale this to handle 10,000 concurrent users?

Session state bottleneck: The in-memory active_sessions dict is on one server. Fix: move session state to Redis with TTL-based expiration. All backend instances share the same Redis cluster.

Horizontal scaling: Run multiple FastAPI instances behind a load balancer (AWS ALB, Nginx). Since state is now in Redis, any instance can serve any session.

Ingestion bottleneck: PDF processing is synchronous in the request handler. Fix: move to a job queue (Celery + Redis, or AWS SQS + Lambda). The /upload endpoint returns a job ID; the frontend polls /status/{job_id}.

Pinecone: Already managed and scales automatically. No changes needed.

Gemini API rate limits: Implement per-user rate limiting (20 queries/hour) using a Redis sliding window counter. Add retry logic with exponential backoff for 429 errors.

Embedding cost at scale: Cache embeddings for repeated queries. Hash the query text + namespace as a Redis key; on cache hit, skip the Gemini embedding call.

Q8How would you handle multi-document Q&A?

Option 1 — Single namespace, multi-document: Allow uploading multiple PDFs to the same session. Tag each chunk's metadata with document_id and filename. At retrieval time, Pinecone returns the most relevant chunks across all documents.

Option 2 — Separate namespaces, routing layer: Each document gets its own namespace. A router LLM determines which namespace(s) to query based on the question. Results are merged and re-ranked before generation.

Challenges: Context window fills up faster when combining multiple documents. Attribution becomes harder. Re-ranking is critical when results come from multiple sources.

For this project, Option 1 is simpler to implement — just remove the "one session = one PDF" constraint and allow multiple uploads under the same session_id.

Q9How would you update the knowledge base when a document changes?

Track document versions: Store a hash (SHA-256) of each document's content in a database. On re-upload, compare hashes. If unchanged, skip re-ingestion. If changed, trigger the update pipeline.

Update pipeline:

  1. Delete all vectors in Pinecone associated with the old document (using metadata filter document_id = X).
  2. Re-ingest the new version: extract, chunk, embed, store.
  3. Update the version hash in the database.

Granular updates: Track which pages changed (compare page-level hashes). Only re-embed the changed pages, not the entire document.

Pinecone caveat: Pinecone doesn't support update-in-place for vectors. You must delete and re-insert.

Q10How would you add source attribution and citations to every answer?

Source attribution is already partially implemented — the backend extracts page_label and file_name from node metadata and returns them in the API response.

To make it production-grade:

  1. Chunk-level IDs: Assign a unique ID to each chunk at ingestion time. Store the ID alongside the chunk in Pinecone metadata.
  2. Citation tracking in generation: After streaming completes, map each claim in the answer to the source chunk — either via post-hoc attribution (ask the LLM) or structured generation (prompt the LLM to output JSON with citations inline).
  3. UI rendering: Render source cards below the answer — showing the page number, a snippet of the retrieved text, and a visual indicator that the answer is grounded.
  4. Faithfulness check: After generation, verify that every sentence in the answer is entailed by at least one retrieved source. Flag or filter answers that fail this check.
Q11How would you implement access control so users can only query their own documents?

Authentication layer: Add JWT-based auth (e.g., using python-jose on the backend, NextAuth.js on the frontend). Every request must include a valid JWT in the Authorization header. The backend extracts user_id from the JWT payload.

Session ownership: Store session_id → user_id mapping in a database (or Redis). On every /chat and /clear request, verify that the JWT's user_id matches the session's owner. If not, return HTTP 403 Forbidden.

Pinecone namespace security: The namespace is currently the session_id. With auth, tie it to user_id:session_id for an extra layer. This ensures that even if the backend code has a bug, a user's namespace is cryptographically tied to their identity.

Additional hardening: Rate limit per user_id to prevent one user from exhausting quotas. Log all access with user_id for auditability.

Debugging & Tradeoffs Questions
Q12A user says the system gave a wrong answer that was actually in the document. What would you investigate?

This is a classic retrieval failure, not a generation failure. The LLM can only answer from what was retrieved.

Step 1: Log and inspect retrieved chunks — Add logging in the /chat endpoint to record which source nodes were retrieved for the query, including their similarity scores. Check: was the chunk containing the correct answer actually retrieved?

Step 2: Diagnose retrieval failure

  • Query phrasing mismatch: The user asked "When was John born?" but the document says "Date of birth: March 1990." Fix: query expansion.
  • Chunk boundary split: The answer spans two chunks. Fix: increase chunk overlap or use sentence window retrieval.
  • Below top-k cutoff: The correct chunk was retrieved at rank 6 but only top-5 are used. Fix: increase k, or add a reranker.

Step 3: Check embedding quality — Try retrieving with a slightly rephrased query. If results change dramatically, the embedding model is sensitive to query wording. Consider HyDE.

Step 4: Check generation — If the correct chunk WAS retrieved, check if the LLM ignored it. Fix: add a faithfulness check, reduce k to reduce noise.

Q13Your retrieval latency increased from 100ms to 800ms after adding 10,000 documents. What would you check?
  • Check 1: Pinecone query latency — Use Pinecone's console to see query latency directly. Check if you've hit the free tier request limits causing throttling.
  • Check 2: Embedding latency — The query must be embedded before Pinecone search. Check Gemini API response times separately. If Gemini is slow, implement embedding caching.
  • Check 3: Network latency — Are the backend (Render) and Pinecone (AWS us-east-1) in the same region? Cross-region calls add 50–150ms.
  • Check 4: Top-k post-processing — With more documents, the LlamaIndex post-retrieval processing may be slower. Profile the Python code to find the bottleneck.
  • Check 5: Index type — Pinecone serverless uses ANN. With 10K vectors, ANN is trivially fast. If using a legacy pod-based index, check the pod capacity.
Q14The LLM is ignoring the retrieved context and answering from its training knowledge. How would you fix this?

This is a prompt engineering and context positioning problem. LLMs have an attention bias — content at the beginning and end of the context receives more attention than content in the middle ("lost in the middle" problem).

Fix 1: Rewrite the system prompt

You MUST answer exclusively from the provided document excerpts below.
If the answer is not in the excerpts, say "I don't know based on the provided document."
Do NOT use your training knowledge for factual claims.

Fix 2: Restructure context positioning — Place retrieved chunks at the beginning of the user message (before the question). The model pays more attention to recent content.

Fix 3: Reduce temperature — Lower temperature (e.g., 0.1) makes the model more deterministic and less likely to drift toward confident training knowledge.

Fix 4: Reduce context noise — If k is large and many retrieved chunks are irrelevant, the model may default to training knowledge. Reduce k or add a reranker.

Fix 5: Add output validation — After generation, run a faithfulness check using an LLM-as-judge or semantic similarity between each answer sentence and the source chunks.

Q15How would you reduce cost by 50% without degrading quality?
  • Cache embeddings for repeated queries (~20–30% savings) — Hash the query text. If the same query is asked again, return the cached embedding instead of calling Gemini.
  • Reduce k (fewer retrieved chunks) — Each chunk added to context increases the prompt token count. If k=5 is current, experiment with k=3. Use retrieval evaluation to confirm quality doesn't degrade.
  • Self-host the embedding model (~40–60% savings) — Replace gemini-embedding-001 with a self-hosted open-source model (BAAI/bge-small-en-v1.5, 33M parameters).
  • Cheaper model for simple queries — Implement query classification: simple factual questions go to a cheaper/faster model; complex synthesis questions go to Gemini 2.5 Flash.
  • Prompt compression — Use LLMLingua to compress the retrieved context before injecting it into the prompt. Compressed context can be 50–70% smaller with minimal information loss.
Project-Specific Questions
Q16Why did you choose Pinecone over a local vector store like FAISS or Chroma?

FAISS: In-memory, requires the entire index to be loaded into RAM on each server start. No persistence by default. No namespace support — building per-user isolation would require separate FAISS indexes, one per user. Impractical for multi-user production.

Chroma: Persistent (disk-based), simpler than FAISS for production use. Has collection-based isolation, but runs as a local service — adds operational complexity on Render. No managed cloud offering at the time of this project.

Pinecone: Fully managed — no infrastructure to operate. Built-in namespace support (perfect for per-user isolation). Persistent across server restarts. Pay-per-query serverless model. Integrates natively with LlamaIndex via PineconeVectorStore.

The main tradeoff: Pinecone has vendor lock-in and free tier limits. For a production system with predictable load, a self-hosted Qdrant or Weaviate instance on a managed Kubernetes cluster would be more cost-effective at scale.

Q17Why Gemini 2.5 Flash instead of GPT-4 or Claude?

Gemini 2.5 Flash advantages for this project:

  • Single API key for both embedding and generation. Using gemini-embedding-001 and gemini-2.5-flash from the same provider eliminates cross-provider complexity.
  • Speed. Flash is optimized for low-latency inference — critical for a streaming chat application where time-to-first-token matters.
  • Cost. Flash is significantly cheaper per token than GPT-4 Turbo or Claude Opus.
  • Context window. Gemini Flash supports up to 1M tokens, eliminating context window concerns for all practical PDF sizes.
  • Free tier. Google Generative AI has a generous free tier that enables zero-cost development and demo deployment.

Tradeoffs: GPT-4 has a larger ecosystem. Claude 3.5 Sonnet may outperform Flash on complex reasoning tasks. Gemini's function calling and structured output support is slightly less mature than OpenAI's.

Q18What would you change if you needed to support 100-page documents?

Remove the 5-page limit, but also address the downstream consequences:

Ingestion time: 100 pages = ~50,000–100,000 tokens of text, producing hundreds of chunks. Embedding all of them takes 10–30 seconds. This will timeout a synchronous HTTP request. Fix: async ingestion with job queues. Return a job ID; poll for completion.

Chunking strategy: Default LlamaIndex chunking (1024 tokens) produces ~100 chunks for 100 pages. Fix: hierarchical chunking — create summary nodes at the chapter/section level alongside fine-grained paragraph nodes. Route queries to the appropriate granularity.

Context window for large results: If a user asks a question about a topic that spans 20 pages, the top-k retrieval may pull too many chunks, exceeding the LLM's context window. Fix: add a summarization step for broad queries, and use reranking to select the most informative chunks.

Memory: Increase ChatMemoryBuffer size or use a "document-aware" memory that re-retrieves relevant earlier context on demand.

Q19Your backend is on Render with cold starts. How would you fix this for production?

Root cause: Render's free tier spins down the container after 15 minutes of inactivity. On the next request, the container must cold-start (download image, start Python, load models, initialize Pinecone connection), which takes 30–60 seconds.

Immediate fixes (cheap):

  • Render paid tier ($7/month): Keeps the container always warm. Eliminates cold starts completely.
  • External ping service: Use UptimeRobot or a GitHub Actions cron job to ping the backend's /health endpoint every 10 minutes. Free but fragile.

Production-grade fixes:

  • Google Cloud Run with min-instances=1: Always keeps one instance warm. Pay only for active requests beyond the minimum instance.
  • Pre-warming: On the frontend, send a lightweight /health ping to the backend when the user first opens the page. By the time they click "Upload," the backend is ready.
Q20Explain how namespace-based multi-tenancy works in Pinecone and why you chose it.

A Pinecone index is a single collection of vectors, all with the same dimensionality and metric (cosine in this project). A namespace is a logical partition within an index. Vectors in different namespaces never interact:

  • Upsert to namespace "session-A" → only queryable in namespace "session-A"
  • Query namespace "session-A" → only returns vectors from namespace "session-A"
  • Namespace "session-B" is completely isolated, despite sharing the same underlying index
# When indexing (upload)
vector_store = PineconeVectorStore(pinecone_index=pinecone_index, namespace=session_id)
# LlamaIndex automatically scopes the query to the same namespace
ApproachProsCons
Namespace per user (current)Single index to manage, no index creation per user, low overheadAll users share index capacity
Index per userComplete isolation, independent scalingPinecone charges per index, slow to create, operational complexity
Metadata filteringNo namespace overheadSlower (scans all vectors, then filters), potential data leakage bugs

Namespaces are the Pinecone-recommended approach for multi-tenant applications. The session_id UUID ensures namespace names are unpredictable, adding an additional layer of access control.

The Hardest Problems I Solved

Sync/Async Bridge

LlamaIndex's ChatEngine is synchronous. Calling it directly inside an async FastAPI handler blocks the entire event loop, making the server unresponsive to concurrent requests. I offloaded it to a thread pool via loop.run_in_executor(), keeping the server non-blocking while the LLM generates.

Multi-User Data Isolation

Each PDF upload generates a UUID that doubles as both the session ID and the Pinecone namespace. This scopes vector retrieval at the database level — not just in application logic — so user A's document embeddings are never reachable from user B's queries.

Token Streaming

LlamaIndex's stream_chat() returns a synchronous generator. I wrapped it in a FastAPI StreamingResponse using an async generator that yields SSE-formatted tokens. The frontend consumes this with the native Fetch API, rendering each token as it arrives.
